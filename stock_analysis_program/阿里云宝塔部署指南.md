# 阿里云宝塔部署指南（前后端一体化 + Nginx 统一入口）

> 本文档已替代旧路线“GitHub Pages 托管前端 + 公网直接暴露 `9000` API”。
>
> **当前唯一推荐的正式部署方案：**
> - 前端和后端一起部署到阿里云服务器
> - Nginx 统一对外提供访问入口
> - 前端走同源 `/api`
> - 后端只监听本机 `9000`

---

## 一、部署目标

部署完成后，系统访问方式应为：

- 首页：`http://101.133.150.164/`
- API：`http://101.133.150.164/api/...`

后续绑定域名并启用 HTTPS 后，应升级为：

- `https://你的域名/`
- `https://你的域名/api/...`

---

## 二、推荐架构

### 1. 角色分工

| 组件 | 职责 | 对外暴露 |
|------|------|----------|
| Nginx | 静态前端托管、`/api/` 反向代理 | 是（80/443） |
| Python 后端 | 运行 `real_data_backend.py` | 否，仅本机 `9000` |
| 前端页面 | `real_data_frontend.html` 及相关静态资源 | 由 Nginx 托管 |
| 数据文件 | `data/positions.json` 等本地持久化数据 | 否 |

### 2. 为什么不再推荐旧路线

旧路线的问题很明确：

- GitHub Pages 默认是 HTTPS
- 云服务器裸 API 常常还是 HTTP
- 浏览器会触发 mixed content 拦截
- 运维上会出现前端、后端、域名、缓存各自分裂的问题

所以正式部署直接收口成一套：**阿里云同机部署 + Nginx 统一入口。**

## 三、服务器与端口规划

### 1. 推荐目录

如果整仓库部署：

```text
/www/wwwroot/Intelligent-stock-analysis-platform/
```

前后端实际目录：

```text
/www/wwwroot/Intelligent-stock-analysis-platform/stock_analysis_program/
```

如果只上传业务目录：

```text
/www/wwwroot/stock_analysis_program/
```

### 2. 端口策略

| 端口 | 用途 | 建议 |
|------|------|------|
| 80 | HTTP 站点入口 | 对公网开放 |
| 443 | HTTPS 站点入口 | 绑定域名后开放 |
| 9000 | Python 后端监听 | 仅服务器本机使用，不长期公网开放 |

## 四、部署前提

- 阿里云服务器已开通
- 宝塔 Linux 面板已安装
- 可登录宝塔管理后台
- 已安装 Nginx
- 已安装 Python 项目管理器
- 已安装 Python 3.10 或 3.11
- 已准备 Tushare Token

## 五、部署实施顺序

### 第 1 步：上传或拉取代码

推荐通过 Git 拉取：

```bash
cd /www/wwwroot
git clone https://github.com/Yanlongyue/Intelligent-stock-analysis-platform.git
```

### 第 2 步：安装 Python 依赖

```bash
cd /www/wwwroot/Intelligent-stock-analysis-platform/stock_analysis_program
python3 -m pip install -r requirements.txt
```

### 第 3 步：配置后端环境变量

通过宝塔 Python 项目管理器或守护进程注入：

```text
TUSHARE_TOKEN=你的Token
```

> Token 只允许通过环境变量注入，不写入代码、仓库、文档或公开配置。

### 第 4 步：启动后端并验证本机接口

```bash
curl http://127.0.0.1:9000/api/health
curl http://127.0.0.1:9000/api/data_status
```

### 第 5 步：新增宝塔网站并托管前端静态文件

站点根目录指向：

```text
/www/wwwroot/Intelligent-stock-analysis-platform/stock_analysis_program
```

### 第 6 步：配置 Nginx 统一入口

项目里已提供样例：

```text
stock_analysis_program/nginx.stock-analysis.conf.sample
```

Nginx 配置完成后，必须满足：

- `/` 打开 `real_data_frontend.html`
- `/api/` 代理到 `127.0.0.1:9000`

### 第 7 步：公网验收

```bash
curl http://101.133.150.164/api/health
```

并在浏览器打开：

```text
http://101.133.150.164/
```

### 第 8 步：收口 9000，准备 HTTPS

当 `/api/` 代理稳定后：

- 关闭公网 `9000`
- 保留 `80/443`
- 有域名后申请 Let's Encrypt 证书并切换 HTTPS

## 七、与代码配置的对应关系

当前项目已经做了这些适配：

- `config.js` 生产环境已切换为 `same-origin`
- 前端线上访问将自动使用当前站点域名作为 API 基础地址
- 不再要求在前端硬编码公网 IP `:9000`

这意味着线上浏览器会请求：

```text
http://当前站点/api/...
```

而不是：

```text
http://101.133.150.164:9000/api/...
```

## 九、常见问题排查

### 问题 1：页面打不开

优先检查：

- 站点根目录是否正确
- `real_data_frontend.html` 是否存在
- Nginx 配置是否已保存并重载

### 问题 2：页面打开了但没数据

优先检查：

- `curl http://127.0.0.1:9000/api/health`
- `curl http://101.133.150.164/api/health`
- `/api/` 反代是否生效
- `TUSHARE_TOKEN` 是否已注入

### 问题 3：还在报 mixed content

优先检查：

- 是否还在用 GitHub Pages 旧地址
- 是否还有地方硬编码 `http://IP:9000`
- 前端缓存是否未刷新

### 问题 4：真实数据异常

优先检查：

- `/api/data_status`
- 当前数据源是否可用
- 后端日志是否报 Token、网络或依赖错误
